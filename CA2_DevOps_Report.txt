1. PROJECT SYNOPSIS (10 Marks)
-----------------------------------------
Project Title: Automated CI/CD Pipeline for MKPDesigns (MERN Stack)
Tools Used: Docker, Jenkins, GitHub Actions
Architecture/Workflow:
- Code Push -> GitHub Repository
- Pull Request -> GitHub Actions (CI: Linting, Testing, Security Audits, Docker Validation)
- Merge to Main -> Webhook triggers Jenkins
- Jenkins (CD) -> Build Docker Images -> Secure Secret Injection -> Deploy Containers
Expected Results: Automated, zero-downtime deployments with consistent Dockerized environments, enforced security scanning, and strict version control.

In-Depth Execution Details (Phase 3B):

Introduction & Objectives:
The primary objective was to eliminate risky, manual deployment processes and enforce strict code quality and security checks. By building a fully automated CI/CD pipeline, the development lifecycle is drastically accelerated while maintaining environment consistency.

Observations & Conclusion:
The integration of GitHub Actions and Jenkins created a highly resilient deployment pipeline. GitHub Actions excelled at providing immediate CI feedback (testing and security audits) for developers directly in the Pull Request UI. Simultaneously, Jenkins proved to be incredibly reliable for the CD process, handling the heavy lifting of securely injecting environment variables and orchestrating the Docker container deployments.

6. CHALLENGES FACED & SOLUTIONS
-----------------------------------------
Challenge 1: Docker-in-Docker (DIND) Constraints
- Issue: The initial Jenkins pipeline failed with `docker-compose: not found` because the Jenkins instance was isolated inside its own container without access to the host's Docker Engine.
- Solution: Restarted the Jenkins container as the `root` user and securely mounted the host machine's Docker socket (`-v /var/run/docker.sock:/var/run/docker.sock`). I then installed the Docker CLI inside Jenkins, bridging the containerized Jenkins to the host Docker daemon.

Challenge 2: Git Workspace Security Mismatches
- Issue: Jenkins threw `fatal: not in a git directory` (Exit Code 128) when checking out the SCM. This occurred because running the Jenkins container as root caused a security mismatch with the git directory ownership.
- Solution: Executed a deep clean of the corrupted Jenkins workspace and globally configured Git inside the container to trust the directory using `git config --global --add safe.directory '*'`.

Challenge 3: Missing `.env` Secrets During Jenkins Deployment
- Issue: The `Deploy Containers` stage failed because `docker-compose up` requires `.env` files, which are securely `.gitignore`d and absent from GitHub. 
- Solution: Built a secure credentials vault on the Jenkins host. I injected the real `.env` variables securely during the pipeline execution (via the `Jenkinsfile`), preventing secrets from leaking into version control while ensuring full application functionality.
